## Codebase Patterns
- Plain Node.js app, no build step. Quality check is `npm run check` (a chain of `node --check` syntax checks). When adding a new lib/*.js file, ALSO add it to the `check` script in package.json.
- Data-access layer lives in lib/. lib/airtable.js is the current Airtable client; lib/domain.js has mappers (mapSailingRecord, escapeFormulaString, normalizeValue) + rate math; lib/config.js reads env via process.loadEnvFile. lib/session.js handles sessions.
- Airtable field names are exact strings like 'Origin Port', 'Destination Port/Via Port', '20D Rate', '40D rate' (lowercase r!), '40HC Rate', 'Rate Effective Date', 'RateView', 'CompanyID', 'MarginPercent', 'DeparturePort'. The SQLite store layer (US-003) must return these exact names so domain.js works unchanged.
- SQLite via node:sqlite `DatabaseSync` needs Node >=22 and emits an ExperimentalWarning at runtime (harmless). require('./lib/db.js') exports `{ db, dbPath }`; db.exec runs multi-statement DDL. lib/db.js creates the data/ dir with fs.mkdirSync before opening. DB path defaults to data/rateninja.db, overridable via SQLITE_DB_PATH. data/*.db is gitignored.
- RateNinja/ is a dirty git submodule pointer (mode 160000) unrelated to Ralph work; do NOT stage it in commits.
- A live `.env` with a real AIRTABLE_PAT exists at repo root; lib/config.js loads it via process.loadEnvFile('.env'). This means scripts that hit Airtable actually work end-to-end when run locally (the real base has ~1554 rates, 3 users, 3 companies, ~23310 sailings). The rates table has NO 'Arrival' field in Airtable — projecting it returns HTTP 422, so always fall back to RATE_FIELDS-only.
- node:sqlite prepared statements: `db.prepare(sql).run(...positionalParams)`. Params must be null/number/bigint/string/Uint8Array — convert booleans to 0/1 yourself. Wrap bulk inserts in db.exec('BEGIN')/COMMIT with ROLLBACK on error.
- Airtable user field for company id is 'CompanyID (from CompanyReference)' (a lookup field name with spaces/parens), distinct from the 'CompanyReference' link field.
- lib/store.js (US-003) is the SQLite read/write layer returning Airtable-shaped `{ id, fields }` records. Pattern: build `fields` with an `assign(fields, name, value)` helper that OMITS null/undefined so domain.js normalizeValue fallbacks ('N/A','') apply. rates 'RateView' is SPLIT on ', ' back into an array (so rateVisibleToView's asArray().some() works); company/user 'RateView' stays a scalar string (compared via sameValue). Boolean cols convert 0/1 -> false/true (companies Admin, users AdminScreen). node:sqlite reads: `db.prepare(sql).all(...params)` (rows array) / `.get(...params)` (single row or undefined).
- getSailings maps rows through domain.mapSailingRecord({ fields }) to produce output identical to the old fetchSailings.
